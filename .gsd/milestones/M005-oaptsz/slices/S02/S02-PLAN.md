# S02: Notification center + per-workflow preferences (in-app channel)

**Goal:** Ship the in-app notification substrate for M005-oaptsz: a `notifications` table, a `notification_preferences` table keyed on (user_id, workflow_id NULL = team-default, event_type), the `app.core.notify(user_id, kind, payload, source_*)` helper that downstream slices and any future workflow engine can call, list/mark-read REST endpoints, a top-bar NotificationBell using the existing `dropdown-menu` primitive with a chronological NotificationPanel and 5s polling, a Notifications tab on the existing `settings.tsx` page for team-default-per-event-type toggles, and notify() wired at the events that exist today (team_invite_accepted at teams.py:245, project_created at projects.py:296, plus a system test trigger gated to system_admin) so the bell renders real content. The per-workflow override path is schema-supported (workflow_id NULL = team-default, specific UUID = override) but its UI ships in whatever future slice introduces a workflow detail page.
**Demo:** User clicks the bell icon in the top bar, sees a chronological list of notifications (today's seed: team invite accepted, project created, plus an admin-triggered system test event) with unread badge count, mark-as-read, mark-all-as-read, and show-only-unread filter. On a Notification Preferences settings panel, the user toggles team-default per-event-type in-app routing (e.g. team_invite_accepted → off; project_created → on); save propagates and subsequent events respect the preferences. With two browser contexts open for the same user, marking a notification read in one context propagates to the other within 5s (5s polling cadence).

ORDER BY created_at DESC LIMIT 20"` for forensic inspection. Frontend exposes a `notifications.poll.tick count=<n>` console.info when ?devtools=1 (MEM341). Failure visibility: `notify()` failures (DB error mid-insert) log ERROR `notify.insert_failed user_id=<uuid> kind=<kind> cause=<exception_class>` and never raise into the calling route — invite acceptance / project creation must not fail because the notification side-effect failed. The notifications list route returns 500 on DB error with the standard FastAPI handler; no silent fallback. Redaction constraints: notification `payload` is JSONB free-form; the helper redacts any key whose lowercase name contains `email`, `token`, `secret`, or `password` from payloads at insert time (replace value with `<redacted>`). Log lines never include payload bodies, only the `kind` discriminator. Preference toggles never log the user's email.
