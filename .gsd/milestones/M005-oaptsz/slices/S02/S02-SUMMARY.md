# S02: Notification center + per-workflow preferences (in-app channel)

**Shipped the in-app notification substrate end-to-end: s07 schema (notifications + notification_preferences), notify() helper with payload redaction and preference gating, REST endpoints, OpenAPI-typed client, NotificationBell + NotificationPanel mounted in _layout with 5s polling, Notifications settings tab, and notify() wired at team_invite_accepted and project_created.**

## What Happened

## What this slice delivered

S02 is the in-app channel layer for M005-oaptsz. It produces the substrate every later slice rides on: a persisted `notifications` table with sensible-defaults preference rows, a single `notify(...)` helper that downstream callers (S03's pywebpush dispatcher, the not-yet-shipped workflow-run engine, today's invite-accept and project-create flows) call once per event, and a top-bar bell that renders new events within 5 seconds across browser contexts. After S02, the bell is real, the preferences page is real, the cross-device read-state contract is real, and the function signature S03 has to fill is frozen.

### T01 — s07 alembic migration + SQLModel models
Created `backend/app/alembic/versions/s07_notifications.py` (down_revision = `s06e_github_webhook_events`). Two tables: `notifications` (UUID PK, user FK CASCADE, kind VARCHAR(64) with CHECK in the seven enum values, payload JSONB DEFAULT '{}', read_at NULL, created_at NOW(), source_team_id FK SET NULL, source_project_id FK SET NULL, source_workflow_run_id UUID with **no FK** because workflow_run table doesn't exist yet — comment explains the deferred FK-add); `notification_preferences` (synthetic UUID id PK + the COALESCE UNIQUE INDEX on (user_id, COALESCE(workflow_id, '00000000-…'::uuid), event_type) for business uniqueness, in_app DEFAULT TRUE, push DEFAULT FALSE). Indexes: (user_id, created_at DESC) and the partial index on (user_id, read_at) WHERE read_at IS NULL for unread_count perf. SQLModel additions in `backend/app/models.py`: `NotificationKind` enum (workflow_run_started/_succeeded/_failed, workflow_step_completed, team_invite_accepted, project_created, system), table classes, and the public DTOs (`NotificationPublic`, `NotificationsPublic`, `NotificationPreferencePublic`, `NotificationPreferencePut`). The migration test `backend/tests/migrations/test_s07_notifications_migration.py` mirrors s01_migration's autouse session-release fixture (commits, expires_all, closes, engine.dispose) per MEM016 to avoid the alembic-DDL deadlock and asserts both tables, the partial unread-count index, the COALESCE uniqueness contract, and downgrade symmetry.

### T02 — notify() helper, REST routes, OpenAPI client regeneration
`backend/app/core/notify.py` exposes `notify(session, *, user_id, kind, payload=None, source_team_id=None, source_project_id=None, source_workflow_run_id=None) -> Notification | None`. Behaviour: (1) resolve effective preference by reading `notification_preferences` for `(user_id, workflow_id IS NULL, event_type=kind.value)`; if no row exists fall back to a hard-coded DEFAULTS dict; (2) on `in_app=False` log `notify.skipped_in_app reason=preference_off` and return None; (3) redact payload by replacing the value of any key whose lowercase name contains 'password', 'token', 'secret', or 'email' with the literal string `<redacted>`, mutating a copy; (4) INSERT and log `notify.dispatched user_id=… kind=… notification_id=… in_app=true`. Wraps the entire body in try/except — DB errors log `notify.insert_failed user_id=… kind=… cause=<exception_class>` ERROR and return None; the helper never re-raises (callers must not fail because of notification side-effects). `_push_stub` is a signature-stable no-op so S03 fills it in without re-touching every call site. `backend/app/api/routes/notifications.py` mounted in `backend/app/api/main.py` exposes GET /notifications, GET /notifications/unread_count, POST /notifications/{id}/read, POST /notifications/read_all, GET /notifications/preferences (merged with DEFAULTS so the UI always sees one entry per kind), PUT /notifications/preferences/{event_type} (UPSERT team-default row, workflow_id NULL). NotificationPublic.kind and NotificationPreferencePublic.event_type are typed as the `NotificationKind` enum (not str) so OpenAPI emits the seven literal values into `frontend/src/client/schemas.gen.ts` (MEM345 — without this the slice contract grep fails). NotificationPreferencePut was slimmed to {in_app, push} since event_type lives on the URL path. OpenAPI regenerated via `bash scripts/generate-client.sh` so the frontend has typed `NotificationsService` access.

### T03 — Wire notify() at team_invite_accepted, project_created, and a system-admin test endpoint
`backend/app/api/routes/teams.py` join_team: snapshot the response body (`team.model_dump()`) BEFORE calling notify(), then call `notify(session, user_id=current_user.id, kind=NotificationKind.team_invite_accepted, payload={'team_id': str(team.id), 'team_name': team.name}, source_team_id=team.id)`, return from the snapshot. Captured as MEM346: notify() commits internally and SQLModel session expiration would empty `team.name` on a post-commit re-serialize, crashing FastAPI's response validator. Same pattern in `backend/app/api/routes/projects.py` create_project — snapshot ProjectPublic, then iterate `recipients = TeamMember.role IN [TeamRole.admin]` and notify each (TeamRole in this codebase is enum {member, admin}; there is no owner — adjusted from the plan's `[admin, owner]`). Both call sites are wrapped in route-level try/except logging `invite_accept_notify_failed` / `project_create_notify_failed` at WARNING for defense in depth. `POST /notifications/test` added to notifications.py, gated on `Depends(get_current_active_superuser)`, accepts `{user_id?, message?, kind?}` and returns the created NotificationPublic — or null body / 200 when notify() returns None due to a preference suppression (later refined in T05).

### T04 — NotificationBell + NotificationPanel + 5s polling, mounted in _layout.tsx
Built using existing primitives (no new deps): `frontend/src/components/notifications/NotificationBell.tsx` is a Button (variant='ghost' size='icon') wrapping a `Bell` lucide icon with an absolute-positioned unread-count badge; the bell is the trigger inside a DropdownMenu and the panel renders inside `DropdownMenuContent`. `NotificationPanel.tsx` shows the title, an 'Unread only' Switch (split into a separate `<label htmlFor>` because biome's noLabelWithoutControl wouldn't allow nesting Radix Switch in a native label), 'Mark all read' Button, and a ScrollArea of `NotificationItem` rows. Each row carries kind icon, human-readable title from kind+payload, relative time, an `<output aria-label='unread'>` dot for the unread state (biome's useSemanticElements rejected `<span aria-label>` and `<span role='status'>`), and onClick → NotificationsService.read(id) + React-Query invalidate. `useNotificationsPolling.ts` returns `{items, unreadCount, isFetching}` from two `useQuery` calls (`['notifications', {unreadOnly}]` and `['notifications','unreadCount']`) both with `refetchInterval: 5000, refetchIntervalInBackground: false`. The unread count is polled independent of panel-open state so the badge stays current. Bell mounted in `_layout.tsx` header inside `<div className='ml-auto flex items-center gap-2'>` so S03's push prompt and S04's mic affordance can land beside it without refactoring; the bell inherits min-h-11/min-w-11 from the design-system-primitive-floor pattern (MEM337). `frontend/tests/m005-oaptsz-notifications.spec.ts` seeds via page.evaluate + dynamic SDK import (MEM347 — the SDK runs in page origin and inherits storageState cookies regardless of FE/API origin split), reloads, asserts the badge appears within 6s, opens the panel, asserts the item, clicks it, asserts the badge clears within 1s.

### T05 — Notifications preferences tab in settings.tsx + cross-device contract gate
Inserted Notifications between Password and Danger zone in `settings.tsx`'s `tabsConfig` so the existing `tabsConfig.slice(0, 3)` admin-widening path naturally includes it. `NotificationPreferences.tsx` renders a Card with a 'team default' note and a table of (Event | In-app | Push) rows — In-app is a Switch wired to `useMutation({ mutationFn: NotificationsService.preferenceUpdate })` whose onSuccess `setQueryData`s the preferences cache directly with the response (immediate optimistic re-anchor) AND `invalidateQueries` on the same key (MEM305 cache-key shape, mirroring PushRuleForm in M004/S06/T04). Push is a disabled Switch labelled 'Available in S03'. POST /notifications/test was refined to return 200 with a null body when notify() is suppressed by a preference (was previously 500 system_channel_suppressed); the null body is the contract signal Scenario B asserts on (MEM350). The contract spec `frontend/tests/m005-oaptsz-notifications-preferences.spec.ts` runs two scenarios: A) two BrowserContexts authenticated as the same user, ContextA seeds → both bells show 1 unread within 6s → ContextA marks read → ContextB's badge clears within 6s; B) toggle team_invite_accepted in_app off → fire kind override → assert no row inserted → toggle back on → fire again → assert the row appears. Both T04's spec and Scenario A target the seeded item's `data-unread` attribute instead of the global badge to survive parallel-worker test execution (MEM349 — parallel workers share the seeded superuser and race on bell badge state).

### What S02 leaves for S03 / S04 / future workflow engine
- S03 fills the empty SW push-event-listener body in `frontend/src/sw.ts`, hydrates the `push_subscriptions` table, wires VAPID generation, and replaces the no-op `_push_stub` in `notify.py` with a real pywebpush dispatcher. The notify() function signature is frozen.
- S04 adds the universal voice input wrapper (independent of S02).
- Per-workflow override UI on a workflow detail page lands when a future milestone ships the workflow run engine. The schema is ready (`workflow_id` is nullable; NULL = team-default, specific UUID = override) and the COALESCE uniqueness contract enforces this without app-level coordination. notify()'s preference resolution is hard-coded to `workflow_id IS NULL` today; widen it when a workflow→workflow_run lookup is wired.
- The `source_workflow_run_id` column has no FK; the future workflow-engine slice should add it.

## Known Limitations

"Per-workflow override UI is not in this slice — the schema supports it (workflow_id NULL = team-default, specific UUID = override; COALESCE UNIQUE INDEX enforces uniqueness without app coordination) but the workflow detail page does not exist yet. notify()'s preference resolution is hard-coded to workflow_id IS NULL today; widen it when workflow→workflow_run lookup is wired by the future workflow-engine slice. POST /notifications/test is a system-admin-gated trigger for UAT/forensic use; production should not depend on it."
