# S01: PWA install + service worker + mobile polish pass

**Goal:** Convert the Perpetuity SPA into an installable PWA on mobile Chrome with a Workbox-injected service worker that correctly bypasses /api/* and /ws/* (so M005-sqm8et's run-status polling is not silently corrupted), ship a Web App Manifest that satisfies Lighthouse install criteria, deliver the install banner + iOS-install-instructions UX, and run a mobile audit + fix pass across every existing flow at 360px and 414px viewport widths so all interactive targets are ≥44×44 CSS pixels with no horizontal scroll.
**Demo:** On a Pixel-class Android device running mobile Chrome, the user opens the app, the install banner appears within 30s, tapping it installs the app, the home-screen icon opens the app standalone, and the user can complete the full existing demo flow (login → dashboard → terminal → projects → run history) without horizontal scroll, with all touch targets ≥44x44 CSS pixels, while the service worker correctly bypasses /api/* and /ws/* requests (proven by an integration test that flips a fixture API response between two SW-mediated fetches and observes the new value).

## Must-Haves

- Valid `frontend/public/manifest.webmanifest` linked from `frontend/index.html`; Lighthouse PWA install criteria pass (manifest validity + SW registration + start_url + icons).
- `frontend/src/sw.ts` registered via vite-plugin-pwa's `injectManifest` strategy; route classifier proven by an integration test that mutates a respx-style fixture between two SW-mediated `/api/v1/*` fetches and observes the fresh value (NetworkOnly bypass is real, not assumed).
- Install banner UX appears on Android Chrome (`beforeinstallprompt` deferred-prompt pattern) with dismiss persisted to localStorage; iOS users see a one-time inline toast with Add-to-Home-Screen instructions.
- Offline banner renders against `navigator.onLine === false` and clears on reconnect (heartbeat to `GET /api/v1/health`).
- Playwright config extends to four projects (`pixel-5-mobile-chrome`, `iphone-13-mobile-safari`, `desktop-chrome`, `desktop-firefox`); a new `tests/m005-oaptsz-mobile-audit.spec.ts` walks every existing flow (auth, dashboard, terminal, projects, GitHub admin, workflow editor, run history, run detail) and asserts no horizontal scroll at 360px + every interactive target ≥44×44 CSS pixels + visual-diff baselines within 1% on critical pages.
- The `m004-guylpp.spec.ts` `testIgnore` list extends to the new mobile-audit spec where appropriate; existing tests still pass on the new project matrix.
- Service-worker dev-mode bypass is a vite-plugin-pwa default — verified by `bun run dev` continuing to serve the app without a SW takeover.

## Verification

- Runtime signals: SW emits `console.info` on `install`, `activate`, `fetch (network-only bypass=/api/*)`, `fetch (cache-hit=/assets/*)`, and `update-available`; main thread emits `pwa.sw.registered`, `pwa.sw.update_available`, `pwa.install.prompt_shown`, `pwa.install.dismissed`, `pwa.install.accepted`, `pwa.offline.detected`, `pwa.online.restored` as `console.info` lines (no remote logging — this is browser-side; a future agent reading the devtools console can localize SW lifecycle bugs).
- Inspection surfaces: `chrome://serviceworker-internals` for SW state, `chrome://inspect/#service-workers` for fetch logs, Application → Manifest tab in DevTools for manifest validity, Application → Service Workers for active registration; `localStorage.pwa.install_dismissed_at` for install-prompt dismissal state.
- Failure visibility: SW registration failure logs `pwa.sw.register_failed reason=<message>` and the app degrades gracefully (no install banner, no offline banner, app still functional).
- Redaction constraints: SW must not log request/response bodies; only URL + method + cache decision (URLs are not secrets but bodies can be — keep the discipline tight to match M002/M004 logging hygiene).
