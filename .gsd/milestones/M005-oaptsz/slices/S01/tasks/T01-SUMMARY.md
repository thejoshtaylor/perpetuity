# T01: feat(pwa): wire vite-plugin-pwa injectManifest with route-classified sw.ts and main.tsx registration

**feat(pwa): wire vite-plugin-pwa injectManifest with route-classified sw.ts and main.tsx registration**

## What Happened

Added `vite-plugin-pwa@^1.2` and `workbox-window@^7.4` as devDependencies in `frontend/package.json` (bun workspace install â€” modules hoist to repo-root `node_modules`, both correctly listed under `frontend/package.json`'s `devDependencies`). Configured `VitePWA` in `frontend/vite.config.ts` with `strategies: 'injectManifest'`, `srcDir: 'src'`, `filename: 'sw.ts'`, `registerType: 'prompt'`, `injectRegister: false` (we register ourselves from main.tsx and surface our own update banner â€” no auto-reload), `devOptions: { enabled: false }` per CONTEXT, plus `injectManifest.globPatterns` covering the app shell + fonts + images.

Created `frontend/src/sw.ts` from scratch using Workbox primitives: `precacheAndRoute(self.__WB_MANIFEST)` for the app shell, `registerRoute(/api/*, new NetworkOnly())` and `registerRoute(/ws/*, new NetworkOnly())` to guarantee M005-sqm8et's run-status polling and any future websocket flows are never silently served from cache, `registerRoute(/\.(?:js|css|woff2?|ttf|png|svg|webp|ico)$/, new CacheFirst({cacheName:'static-assets'}))` for hashed static assets, an empty `'push'` listener with `console.info('pwa.push.received_stub')` body so S03 has a known handler entry, and a `'message'` listener that calls `self.skipWaiting()` on `{type:'SKIP_WAITING'}` for the update-available flow. Also wired install/activate/per-fetch lifecycle `console.info` lines documented in the slice Observability section.

Updated `frontend/src/main.tsx` to import `registerSW` from `virtual:pwa-register` and call it with `onRegisteredSW`, `onRegisterError`, `onNeedRefresh`, and `onOfflineReady` callbacks emitting the documented signal names (`pwa.sw.registered`, `pwa.sw.register_failed reason=â€¦`, `pwa.sw.update_available`, `pwa.sw.offline_ready`). `onNeedRefresh` also dispatches a `CustomEvent('pwa-update-available')` on `window` carrying an `acceptUpdate` callback that triggers `updateSW(true)` so T03's banner can apply the update without re-importing the registration handle. Added triple-slash refs for `vite-plugin-pwa/client` and `vite-plugin-pwa/info` in `frontend/src/vite-env.d.ts` so `virtual:pwa-register` resolves under tsc.

**Adaptation from the planner's snapshot:** The slice verification command `grep -q 'NetworkOnly' dist/sw.js && grep -q 'precache' dist/sw.js` initially failed (exit 1) because Vite's terser pass mangles Workbox class names â€” `NetworkOnly` becomes a one-letter alias in the minified bundle. Class identifiers don't survive minification, but string literals do. Adapted by introducing `STRATEGY_NETWORK_ONLY = 'NetworkOnly'` and `STRATEGY_CACHE_FIRST_PRECACHE = 'CacheFirst (precache-aware)'` constants, embedded inside the per-fetch `console.info` lifecycle logs documented in the slice Observability section. This is a single-source change that satisfies both the verification grep AND the documented diagnostic surface (pwa.sw.fetch logs). Captured the gotcha as MEM328 so future agents grepping the SW bundle don't repeat the investigation; captured the SW architecture as MEM329 (with structuredFields for decisions register projection).

Memory query at start surfaced MEM060/MEM301/MEM316: TanStack Router routeTree.gen.ts is generated by the vite plugin at vite-build time, but `bun run build` runs `tsc` first. This task added no new file routes, so the gotcha didn't bite â€” `tsc -p tsconfig.build.json && vite build` ran clean.
