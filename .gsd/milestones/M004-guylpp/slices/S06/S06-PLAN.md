**Goal:** Ship the M004 admin + team-admin frontend: SystemSettings UI for GitHub App credentials with sensitive-key visualization and one-time-display generate modal; per-team GitHub connections list with install CTA and uninstall; per-team projects list with create, open, and push-rule form (all three modes; rule + manual_workflow tagged "executor lands in M005"); team-mirror always-on toggle. Prove the full admin-side experience end-to-end with a Playwright spec that walks install (mocked GitHub callback), generate-secret modal one-time-display, project create + open (mocked clone path), push-rule persistence across all three modes, and mirror always-on toggle — all four flows complete without hitting the real GitHub API.
**Demo:** Playwright e2e walks the full admin-side experience: install GitHub App (mocked GitHub callback returns installation_id), create a project, open it (mocked clone path), configure a push rule, generate the webhook secret and confirm one-time-display modal, toggle mirror always-on. All four flows complete without hitting the real GitHub API.

## Must-Haves

- After this: the regenerated frontend client carries every M004 endpoint shipped in S01–S05 (admin/settings + generate, github install-url + callback + installations list/delete, projects CRUD + push-rule + open, mirror always-on PATCH); a system_admin lands on `/admin/settings`, sees the four registered GitHub App keys with lock icons + Set/Replace/Generate actions, can paste a synthetic PEM into github_app_private_key (PUT), can click Generate on github_app_webhook_secret and see the value exactly once in a one-time-display modal that explicitly warns the value will not be shown again; a team-admin on `/teams/<id>` sees a Connections section with an "Install GitHub App" CTA that opens the signed-state install URL in a new tab, an installations list (account_login + account_type), and a destructive Uninstall action; a team-admin sees a Projects section with create-project (repo_full_name + installation_id picker), an Open button per project that calls POST /projects/{id}/open and toasts errors with the orchestrator's `reason` field, and a push-rule form with three modes where rule + manual_workflow render a "stored — executor lands in M005" badge; a team-admin sees an always-on toggle for the team mirror that PATCHes /api/v1/teams/{id}/mirror; the consolidated Playwright spec `frontend/tests/m004-guylpp.spec.ts` exercises all four flows against the live compose stack with mock-github sidecars replacing api.github.com and a git-daemon clone target — passes green and surfaces no console errors.
- ## Threat Surface
- **Abuse**: New install-CTA opens window via `window.open(url, '_blank', 'noopener')` — `noopener` is mandatory to prevent the target tab from manipulating window.opener. Generate-confirm modal must require an explicit click (not Enter-on-form) so an in-page XSS cannot drive a passive secret regeneration. Uninstall and mirror-toggle endpoints are already team-admin-gated server-side; UI must not weaken that by relying on UI-only visibility.
- **Data exposure**: github_app_webhook_secret plaintext crosses the FE boundary EXACTLY ONCE, only inside the OneTimeValueModal. No React state holds the value past modal close; no console.log; no analytics; no localStorage. Copy-to-clipboard is via navigator.clipboard.writeText only — never an inline DOM-readable text node persisted after close. PEM input field must be a textarea with autocomplete=off and not echoed back in the PUT response toast.
- **Input trust**: PEM textarea (github_app_private_key) and webhook-secret PUT alternative are user-provided strings POSTed verbatim to the backend, which already runs the structural PEM validator and Fernet-encrypts before storage (S01). repo_full_name on Create Project must be a single-line input — backend already normalizes — but FE should reject literal newlines client-side to keep error toast quality high. branch_pattern / workflow_id on push-rule are stored as opaque strings (M005 will tighten); reject only empty after trim.

## Integration Closure

Each FE component is wired to a real backend endpoint by URL pattern (no fixture data, no MSW handlers) and proven by the Playwright spec running against the live compose stack. The install flow's "redirect to github.com" hop is closed by a sibling helper container that intercepts the install URL and POSTs the callback as if GitHub had redirected the user back — the test asserts the resulting installation row appears in the Connections list. The "Open project" flow is closed by triggering the orchestrator's POST /v1/projects/{id}/materialize-{mirror,user} chain through the backend route and asserting the project's status updates in the UI; the underlying clone hits the git-daemon mock-github sidecar (same image used in S04/T05). The Generate-secret one-time-display contract is closed by asserting the plaintext value appears in the modal exactly once, then asserting a subsequent /admin/settings list shows `has_value:true` with `value:null`. The Always-On toggle is closed by PATCH /api/v1/teams/{id}/mirror returning the new flag and the toggle reflecting it. No flow uses MSW, no flow stubs network at the browser layer.

Upstream surfaces consumed: backend admin/settings + generate (S01), github install-url/callback/list/delete (S02), projects CRUD + push-rule + open (S04), teams mirror PATCH (S03→S04).
