# S02: Per-team GitHub connections (install flow + installation tokens)

**Per-team GitHub-App install handshake ships end-to-end (signed-state install-url → public install-callback → idempotent persistence) and the orchestrator can mint and cache installation tokens (RS256 App JWT → /access_tokens → Redis 50-min cache), all with token plaintext kept out of every backend + orchestrator log line.**

## What Happened

S02 closes the credential plumbing the rest of M004 will sit on. Four tasks, in four passes:

T01 created alembic revision `s06b_github_app_installations` (down_revision=`s06_system_settings_sensitive`) plus the `GitHubAppInstallation` SQLModel + `GitHubAppInstallationPublic` projection. Schema: UUID PK, BIGINT `installation_id` UNIQUE, team FK with ON DELETE CASCADE, VARCHAR(255) `account_login`, VARCHAR(64) `account_type` with a CHECK constraint pinning it to `{Organization, User}`, server-defaulted TIMESTAMPTZ `created_at`. Six migration tests cover the upgrade shape, UNIQUE violation, CHECK violation on `account_type='Bot'`, parent-team cascade, downgrade-drop, and a downgrade→upgrade round-trip schema-byte-identity snapshot.

T02 added `backend/app/api/routes/github.py` and registered it in `app/api/main.py`. Four endpoints: GET install-url (team-admin; HS256 state JWT with aud='github-install'/iss='perpetuity-install'/jti/exp=10m signed by SECRET_KEY; returns `{install_url, state, expires_at}` shaped `<GITHUB_APP_INSTALL_URL_BASE>/apps/<client_id>/installations/new?state=<jwt>`); public POST install-callback (no auth dep — the signed state IS the auth; decoded with audience+algorithms pinned; calls orchestrator GET /v1/installations/{id}/lookup with `X-Orchestrator-Key` over a 10s httpx timeout; lookup failure modes shaped per the slice plan as 502 `{detail:'github_lookup_failed', reason:<status|'timeout'|'transport'|'malformed_lookup_response'>}`; UPSERT keyed on installation_id so duplicate callbacks are 200-idempotent; team-reassignment logs WARNING `github_install_callback_team_reassigned`); team-admin GET list ordered by created_at DESC; team-admin DELETE that returns 404 byte-identical for both missing rows AND rows owned by a different team (no cross-team enumeration). Settings gained `GITHUB_APP_INSTALL_URL_BASE` (default https://github.com). 27 unit tests in `test_github_install.py` — install-url state shape + signature + 10m expiry, install-url 404 when client_id unset, install-callback happy path with `_FakeAsyncClient` stubbing the orchestrator lookup, idempotent duplicate-installation_id, expired-state 400, bad-signature 400, wrong audience 400, team_unknown 400, orchestrator 503/timeout/non-JSON/missing-keys → 502 with the expected reason and zero rows persisted, list ordered + empty envelope + non-admin 403, delete 404 paths, and a redaction sanity check asserting the full state JWT never appears in any captured log record.

T03 added `orchestrator/orchestrator/github_tokens.py` and `routes_github.py`. `_load_github_app_credentials(pg_pool)` reads both system_settings rows in a single batched SELECT, decrypts the private key via the existing Fernet substrate, and re-attaches the row key on `SystemSettingDecryptError` so the global handler logs `key=github_app_private_key`. Missing rows / NULL value_encrypted surface as internal `_NotConfigured` (route layer maps to 503 `github_app_not_configured`). `_mint_app_jwt` uses `jwt.encode(algorithm='RS256')` with claims `{iat=now-60, exp=now+540, iss=str(app_id)}` — the 60s clock-skew backdate keeps a slow-clock orchestrator inside GitHub's 10-minute App JWT window. `mint_installation_token` POSTs `<github_api_base_url>/app/installations/{id}/access_tokens` with `Authorization: Bearer <jwt>`, parses `{token, expires_at}`, and logs `installation_token_minted installation_id=<id> token_prefix=<first4>...`. Non-2xx → `InstallationTokenMintFailed(status, reason)` where `reason` is whitelist-sanitized via `_short_error_label` (alnum + ` ._-:`) so a hostile GitHub message can never sneak control characters into the log. `get_installation_token` is the cache-first wrapper: cache key `gh:installtok:{id}`, TTL 3000s (50 min). Hit → log `installation_token_cache_hit` and return `source='cache'` with a TTL-derived ISO expiry. Miss → mint → SETEX. Redis unreachable on GET or SETEX is non-fatal: warn `redis_unreachable op=installation_token_<get|setex>` and proceed without caching. `lookup_installation` mirrors the same auth shape against `/app/installations/{id}` and surfaces `{account_login, account_type}` for the install-callback. Routes mounted at `/v1/installations/{id}/{token,lookup}` and gated by the existing `SharedSecretMiddleware`. `_redis_client_from(request)` reuses the existing `RedisSessionRegistry._client` (same pool, separate keyspace prefix). Errors map: `_NotConfigured` → 503 `github_app_not_configured`; `InstallationTokenMintFailed` → 502 `github_token_mint_failed`/`github_lookup_failed` carrying `{status, reason}`; `SystemSettingDecryptError` → 503 `{detail, key}` via a NEW global handler in `main.py` mirroring `backend/app/main.py` exactly. Config gained `github_api_base_url: str = "https://api.github.com"` (env `GITHUB_API_BASE_URL`); pyproject gained `pyjwt[crypto]>=2.8.0,<3.0.0` plus `respx` and `fakeredis` in dev deps. 21 unit tests cover JWT shape (RS256, iss=str(app_id), exp-iat==600s), Bearer-auth-on-outbound, fakeredis cache-hit/cache-miss with TTL exactly 3000s, 401/404 surface as InstallationTokenMintFailed, malformed body fallback paths, missing/NULL/malformed credentials → _NotConfigured, decrypt failure propagates uncaught, lookup happy + malformed body, and a TestClient-driven route surface (200/502/503/200) including a `_BrokenRedis` test that proves Redis-unreachable on GET still mints.

T04 wrote `backend/tests/integration/test_m004_s02_github_install_e2e.py` plus `fixtures/mock_github_app.py` to exercise the full install + token-mint + cache + decrypt-failure contract end-to-end against the live compose db/redis with a sibling mock-github sidecar replacing api.github.com. Topology: mock-github (python:3.12-slim, fastapi/uvicorn/pyjwt installed inline at boot) on perpetuity_default with --network-alias mock-github-<uuid>; ephemeral orchestrator with `GITHUB_API_BASE_URL=http://mock-github-<uuid>:8080` and --network-alias orchestrator (so it owns the compose DNS name); sibling backend pointed at `http://orchestrator:8001` with the test's randomly-generated `ORCHESTRATOR_API_KEY`. Skip-guards probe both baked images for the s06b alembic revision (backend) and the github_tokens.py module (orchestrator). Six scenarios assert: (A) install-URL state JWT shape — admin signup → seed credentials → team-admin signup → GET install-url → in-test HS256-decode against SECRET_KEY validates audience/team_id/exp window; (B) install-callback round-trip with installation_id=42 → 200 with account_login='test-org' account_type='Organization'; list returns 1 row; (C) duplicate install-callback → 200 idempotent, list still 1 row; (D) token mint + cache via `docker exec` urllib probe with X-Orchestrator-Key — first call returns `source='mint' token=<MOCK_FIXED_TOKEN>`, second returns `source='cache'` same token; Redis introspection via `redis-cli -a <pw>` shows exactly one `gh:installtok:42` key with TTL in (1, 3001]; (E) expired state token → 400 `install_state_expired`; (F) decrypt failure — UPDATE system_settings SET value_encrypted = E'\\xdeadbeef' WHERE key='github_app_private_key' (the plan's `E'\\x00bad'` was rejected by psql due to odd hex digit count — see MEM259), flush the install-token cache, hit /token → 503 `system_settings_decrypt_failed key='github_app_private_key'` with the matching ERROR log line. Final redaction sweep over backend + ephemeral-orchestrator logs (mock-github logs explicitly excluded since they contain the canned token by design): zero MOCK_FIXED_TOKEN matches, zero `gho_/ghu_/ghr_/github_pat_` matches, `ghs_` permitted only inside `token_prefix=` lines, zero embedded PEM-body sentinel matches, zero `-----BEGIN` matches. Required positive markers asserted in the same blob: `github_install_url_issued`, `github_install_callback_accepted`, `installation_token_minted`, `installation_token_cache_hit`, `system_settings_decrypt_failed key=github_app_private_key`. 22-23s wall-clock against a warm compose stack (well under the 180s slice budget).

What this slice establishes for downstream slices: S03 (team-mirror container) and S04 (clone+push) consume `get_installation_token(installation_id)` from orchestrator/github_tokens.py for env-on-exec credentials at the git-op call site only. S03's mirror containers will set `GITHUB_API_BASE_URL` in their orchestrator environment via the same config knob landed here. S05 (webhook receiver) is independent of S02. S06 (frontend) consumes the four backend endpoints + the openapi-generated client. The mock-github sidecar pattern (MEM261) is reusable for S04's clone/push and S07's pre-real-GitHub regression sweep.
