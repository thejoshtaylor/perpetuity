# T04: Auto-push executor: post-receive hook deployed at clone-time + orchestrator callback that pushes to GitHub origin with fresh installation token

Wire the push-back loop. Two tightly-coupled pieces, both new:

(a) `orchestrator/orchestrator/clone.py::_install_post_receive_hook(docker, *, mirror_container_id, project_id, push_rule_mode)` — invoked by `clone_to_mirror` AFTER the clone succeeds and the .git/config is sanitized. If `push_rule_mode == 'auto'`, writes a tiny shell script to `/repos/<project_id>.git/hooks/post-receive` inside the mirror container with mode 0755. The script contents (kept minimal so it's auditable from `docker exec <mirror> cat /repos/<project_id>.git/hooks/post-receive`):

  #!/bin/sh
  # Auto-generated by perpetuity orchestrator (M004/S04). DO NOT EDIT.
  exec wget -qO- --post-data='' \
    --header="X-Orchestrator-Key: $PERPETUITY_ORCH_KEY" \
    "http://orchestrator:8001/v1/projects/$PERPETUITY_PROJECT_ID/auto-push-callback" \
    > /dev/null 2>&1

The `PERPETUITY_PROJECT_ID` and `PERPETUITY_ORCH_KEY` env vars are baked into the mirror container's environment when ensure_team_mirror starts it (T04 amends `_build_team_mirror_container_config` in `team_mirror.py` to add ORCHESTRATOR_API_KEY from settings to the mirror container's env, and PERPETUITY_PROJECT_ID is set per-project at install-hook time via a per-project script wrapper). To keep the hook generic across multiple projects in the same mirror, the hook reads `$GIT_DIR` (set by git-daemon to `/repos/<project_id>.git`) and derives project_id from the basename — that way one identical hook script works for every auto-rule project on the team. Refined hook:

  #!/bin/sh
  PROJECT_ID=$(basename "$GIT_DIR" .git)
  exec wget -qO- --post-data='' \
    --header="X-Orchestrator-Key: $PERPETUITY_ORCH_KEY" \
    "http://orchestrator:8001/v1/projects/$PROJECT_ID/auto-push-callback" \
    > /dev/null 2>&1 || true

The trailing `|| true` keeps post-receive from failing the user's push if the orchestrator is briefly unreachable (auto-push is best-effort by D024 — failure surfaces in `projects.last_push_status`). For `push_rule_mode in ('rule', 'manual_workflow')`, no hook is installed (the executors land in M005). For PUT push-rule that flips a project from manual_workflow → auto, T04 also adds an orchestrator endpoint `POST /v1/projects/{project_id}/install-push-hook` that the backend's `PUT /api/v1/projects/{project_id}/push-rule` calls when transitioning to auto, and `POST /v1/projects/{project_id}/uninstall-push-hook` for the reverse — both are no-op if the mirror container doesn't currently exist for the team (the next ensure-clone will install/skip per the persisted rule).

(b) `orchestrator/orchestrator/auto_push.py::run_auto_push(docker, pool, *, project_id) -> {result, exit_code, stderr_short}` — the callback handler. Loads the project from pg (team_id, installation_id, repo_full_name), confirms `project_push_rules.mode == 'auto'` (defensive — rule could have flipped between hook install and invocation), mints a fresh installation token via `get_installation_token`, then docker-execs `git --git-dir=/repos/<project_id>.git push --all --prune https://x-access-token:<TOKEN>@github.com/<repo_full_name>.git` inside the mirror container with the token in env vars only (NEVER logged, NEVER persisted). Also pushes tags via a separate `git push --tags ...` call (`--all` doesn't include tags). On success: UPDATE `projects.last_push_status='ok'`, `last_push_error=NULL`. On non-zero exit: UPDATE `last_push_status='failed'`, `last_push_error=<first 500 chars of stderr, scrubbed>`, log WARNING `auto_push_rejected_by_remote project_id=<uuid> exit_code=<n> stderr_short=<safe>`. The stderr-scrub strips any `gho_/ghs_/ghu_/ghr_/github_pat_` substring before persisting (defense-in-depth). If the push-rule has flipped to non-auto since the hook was installed, return early with `{result:'skipped_rule_changed'}` and log INFO `auto_push_skipped project_id=<uuid> reason=rule_changed`. Adds `POST /v1/projects/{project_id}/auto-push-callback` to `routes_projects.py` (gated by SharedSecretMiddleware — the mirror container's hook presents the X-Orchestrator-Key it inherited from the mirror's env). Tests: 10+ unit tests for `run_auto_push` covering happy-path push, push-rejected-by-remote, rule-changed-skip, missing-installation 404 mapping, token-mint-failure handling, stderr scrubbing of token substrings, last_push_status updates on both branches; route-level tests covering shared-secret enforcement, project-not-found 404, request body parsing.

Backend changes: extend `PUT /api/v1/projects/{project_id}/push-rule` in `routes/projects.py` to call orchestrator's install-push-hook / uninstall-push-hook on rule transitions to/from auto; failures are logged WARNING but DO NOT fail the PUT (the hook can be reinstalled on next clone-to-mirror — the rule is the source of truth, the hook is derived state). Update T01's `test_projects.py` to assert the install/uninstall calls are made in the right transitions (httpx mocked).
