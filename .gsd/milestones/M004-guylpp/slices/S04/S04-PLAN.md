# S04: Projects CRUD + two-hop clone + push-back rule storage + auto-push executor

**Goal:** Stand up the projects domain end-to-end: per-team projects linked to a real GitHub repo (via the team's GitHub App installation), a two-hop materialize path (GitHub→mirror with env-on-exec installation token, then mirror→user over `git://team-mirror-<first8>:9418` with zero credentials in the user container), and an auto-push executor that pushes user commits back to GitHub when the project's push rule is `mode=auto` — with the rule schema landing for all three modes (auto, rule, manual_workflow) but only `auto` wired to an executor in M004. This is the slice that makes M004's primary user loop (R009 + R010) actually work.
**Demo:** Team admin POST /api/v1/teams/{team_id}/projects creates a project linked to a real GitHub repo via installation_id. User POST /api/v1/projects/{id}/open materializes the repo: orchestrator clones GitHub→mirror with installation token (env-on-exec only — verified by inspecting the mirror's /repos/<project_id>.git/config showing bare github.com URL with no token), then user container runs git clone git://team-mirror-<team_id>:9418/<project_id>.git → repo lands at /workspaces/<u>/<t>/<project_name> with no credentials in user-side .git/config. User commits + git push → mirror receives → if rule.mode=auto, mirror auto-pushes to GitHub origin → fixture upstream sees the new ref. PUT /api/v1/projects/{id}/push-rule persists all three modes (auto, rule, manual_workflow); rule and manual_workflow are stored but inert.

## Must-Haves

- Team admin POST /api/v1/teams/{team_id}/projects with `{installation_id, github_repo_full_name, name}` creates a project row and a default `project_push_rules` row (mode=manual_workflow); GET lists team projects; PATCH/DELETE work with team-admin gating; PUT /api/v1/projects/{id}/push-rule persists all three modes (auto, rule, manual_workflow) and rejects unknown modes with 422.
- Team user POST /api/v1/projects/{id}/open returns 200 with the materialized workspace path; backend chains orchestrator `POST /v1/projects/{id}/materialize-mirror` (idempotent) then `POST /v1/projects/{id}/materialize-user`; the user's repo lands at `/workspaces/<user_id>/<team_id>/<project_name>` inside the user-session container; `git -C /repos/<project_id>.git/config remote.origin.url` inside the mirror is bare `https://github.com/<owner>/<repo>.git` (no token); the user container's `.git/config` shows `git://team-mirror-<first8>:9418/<project_id>.git` (no credentials).
- User commits + git push from inside the user container reaches the mirror container's bare repo; if the project's push rule is `mode=auto`, the mirror's `post-receive` hook calls back to the orchestrator and the orchestrator runs `git push origin --all --tags` from inside the mirror container with a freshly-minted installation token in env vars; the fixture upstream (acting as GitHub) sees the new ref with a 0 exit code; on success `projects.last_push_status='ok'`; on failure `projects.last_push_status='failed'` + `last_push_error` is populated.
- Required INFO log keys all observed in a clean run: `team_mirror_clone_started`, `team_mirror_clone_completed`, `user_clone_started`, `user_clone_completed`, `mirror_push_started`, `mirror_push_completed`, `auto_push_started`, `auto_push_completed`. WARNING: `auto_push_rejected_by_remote` on a forced-fail path. No GitHub token plaintext or `gho_/ghs_/-----BEGIN` substrings appear in captured backend or orchestrator logs (per-slice redaction sweep using the same grep set S07 will extend).
- User-session containers are attached to the `perpetuity_default` compose network so they can resolve `team-mirror-<first8>:9418` by DNS (closes MEM264 — without this fix the user-side `git clone git://...` would fail with `Could not resolve host`).

(a) `orchestrator/orchestrator/clone.py::_install_post_receive_hook(docker, *, mirror_container_id, project_id, push_rule_mode)` — invoked by `clone_to_mirror` AFTER the clone succeeds and the .git/config is sanitized. The script contents (kept minimal so it's auditable from `docker exec <mirror> cat /repos/<project_id>.git/hooks/post-receive`):

  #!/bin/sh
  # Auto-generated by perpetuity orchestrator (M004/S04). DO NOT EDIT.
  exec wget -qO- --post-data='' \
    --header="X-Orchestrator-Key: $PERPETUITY_ORCH_KEY" \
    "http://orchestrator:8001/v1/projects/$PERPETUITY_PROJECT_ID/auto-push-callback" \
    > /dev/null 2>&1

The `PERPETUITY_PROJECT_ID` and `PERPETUITY_ORCH_KEY` env vars are baked into the mirror container's environment when ensure_team_mirror starts it (T04 amends `_build_team_mirror_container_config` in `team_mirror.py` to add ORCHESTRATOR_API_KEY from settings to the mirror container's env, and PERPETUITY_PROJECT_ID is set per-project at install-hook time via a per-project script wrapper). To keep the hook generic across multiple projects in the same mirror, the hook reads `$GIT_DIR` (set by git-daemon to `/repos/<project_id>.git`) and derives project_id from the basename — that way one identical hook script works for every auto-rule project on the team. Refined hook:

  #!/bin/sh
  PROJECT_ID=$(basename "$GIT_DIR" .git)
  exec wget -qO- --post-data='' \
    --header="X-Orchestrator-Key: $PERPETUITY_ORCH_KEY" \
    "http://orchestrator:8001/v1/projects/$PROJECT_ID/auto-push-callback" \
    > /dev/null 2>&1 || true

The trailing `|| true` keeps post-receive from failing the user's push if the orchestrator is briefly unreachable (auto-push is best-effort by D024 — failure surfaces in `projects.last_push_status`). For `push_rule_mode in ('rule', 'manual_workflow')`, no hook is installed (the executors land in M005). For PUT push-rule that flips a project from manual_workflow → auto, T04 also adds an orchestrator endpoint `POST /v1/projects/{project_id}/install-push-hook` that the backend's `PUT /api/v1/projects/{project_id}/push-rule` calls when transitioning to auto, and `POST /v1/projects/{project_id}/uninstall-push-hook` for the reverse — both are no-op if the mirror container doesn't currently exist for the team (the next ensure-clone will install/skip per the persisted rule).
