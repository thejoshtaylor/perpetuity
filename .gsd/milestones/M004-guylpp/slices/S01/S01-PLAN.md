# S01: Sensitive system_settings + GitHub App credentials registered

**Goal:** Sensitive system_settings storage exists end-to-end: an admin can paste the GitHub App private key (PEM) once via PUT and subsequent GETs return `has_value:true, value:null`; an admin can POST `/admin/settings/github_app_webhook_secret/generate` to seed a server-side `secrets.token_urlsafe(32)`, see the value exactly once in the response, and never again on subsequent GETs; orchestrator-side `decrypt_setting()` round-trips the encrypted payload back to plaintext at the call site only; corrupted Fernet ciphertext at any decrypt site surfaces as 503 with `system_settings_decrypt_failed` ERROR log naming the key and never silently falls back.
**Demo:** Admin pastes GitHub App private key (PEM) once; GET shows has_value:true, value:null. Admin clicks Generate webhook secret → response shows the secret exactly once → subsequent GET shows has_value:true with no value. Orchestrator-side decrypt_setting() round-trips the private key to plaintext at the call site only. Fernet decrypt failure on a corrupted ciphertext returns 503 with system_settings_decrypt_failed ERROR log.

## Must-Haves

- `_VALIDATORS` is `dict[str, {validator, sensitive: bool, generator: Callable | None}]` with four registered keys: `github_app_id` (public int), `github_app_client_id` (public str), `github_app_private_key` (sensitive PEM, no generator), `github_app_webhook_secret` (sensitive, generator=`secrets.token_urlsafe(32)`).
- `backend/app/core/encryption.py` exposes `encrypt_setting(plaintext: str) -> bytes` and `decrypt_setting(ciphertext: bytes) -> str` (Fernet, key from `SYSTEM_SETTINGS_ENCRYPTION_KEY`).
- App boot raises at module import when any sensitive key is registered and `SYSTEM_SETTINGS_ENCRYPTION_KEY` is missing or not a 32-byte url-safe base64 string.
- `system_settings` table has three new columns: `value_encrypted BYTEA NULL`, `sensitive BOOLEAN NOT NULL DEFAULT FALSE`, `has_value BOOLEAN NOT NULL DEFAULT FALSE`. The legacy `value JSONB` column stays for non-sensitive keys (back-compat with M002 keys).
- `GET /admin/settings/{key}` for sensitive keys returns `{value: null, has_value: <bool>, sensitive: true}`; non-sensitive keys are unchanged.
- `PUT /admin/settings/{key}` for sensitive keys validates plaintext, encrypts, stores into `value_encrypted` with `has_value=true`, never writes JSONB `value`.
- `POST /admin/settings/{key}/generate` returns 422 on keys without a generator; returns 200 `{key, value, has_value: true, generated: true}` on first call and stores the encrypted ciphertext; subsequent PUTs/generates are not blocked but are intentionally destructive (last write wins).
- Decrypt failure (`cryptography.fernet.InvalidToken`) at any call site emits structured ERROR log `system_settings_decrypt_failed key=<name>` and surfaces as 503 `{detail: 'system_settings_decrypt_failed', key: '<name>'}`.
- Orchestrator-side `decrypt_setting` callable mirror exists and is import-clean for S02 to consume.
- e2e `backend/tests/integration/test_m004_s01_sensitive_settings_e2e.py` exercises: register PEM via PUT → GET shows redacted → generate webhook secret → second GET shows redacted → corrupt the stored ciphertext directly in the DB → next decrypt-touching path returns 503 with the named key in detail and matching ERROR log line.

## Proof Level

- This slice proves: contract — slice proves the storage + retrieval + decrypt-failure surface in isolation; S02 will be the first consumer that round-trips the private key for a real GitHub JWT signing call.

## Integration Closure

- Upstream surfaces consumed: M002/S03's `system_settings` table, M002/S05's `_VALIDATORS` registry pattern in `backend/app/api/routes/admin.py`, M002's `get_current_active_superuser` gating, the existing alembic chain head (`s05_system_settings`).
- New wiring introduced in this slice: `cryptography` library dependency added to `backend/pyproject.toml` and `orchestrator/pyproject.toml`; `SYSTEM_SETTINGS_ENCRYPTION_KEY` env var threaded through `docker-compose.yml` (backend + orchestrator services), the e2e conftest's sibling-backend fixture, and the M002 two-key-rotation test's `_boot_sibling_backend` helper if it boots a backend; `app/core/encryption.py` and `orchestrator/orchestrator/encryption.py` modules (mirror — same Fernet helper, distinct config surfaces); migration `s06_system_settings_sensitive.py` extends the table without breaking M002's reads of the existing `value` column.
- What remains before the milestone is truly usable end-to-end: S02 will be the first true consumer (orchestrator imports `decrypt_setting` to sign GitHub App JWTs); S05's webhook receiver will be the second consumer (HMAC compare against decrypted webhook secret). Neither lands here — this slice is contract-only. The orchestrator-side mirror is included so S02 has a clean import target without re-touching S01 plumbing.

## Verification

- Runtime signals: INFO `system_setting_updated key=<name> sensitive=<bool> previous_value_present=<bool>` (extends M002 line with `sensitive` field); INFO `system_setting_generated key=<name> actor_id=<uuid>`; ERROR `system_settings_decrypt_failed key=<name>` (single source of truth for decrypt failures, raised from a single helper so call sites can't drift).
- Inspection surfaces: `GET /admin/settings` lists all keys with `sensitive`, `has_value`, never plaintext; psql `SELECT key, sensitive, has_value, length(value_encrypted) FROM system_settings` is the operator-side state inspector.
- Failure visibility: 503 response body and ERROR log both name the key but never the plaintext; bad SYSTEM_SETTINGS_ENCRYPTION_KEY at boot raises immediately so the failure is at startup, not first-decrypt-call.
- Redaction constraints: NEVER log raw values for sensitive keys, never include the plaintext in any HTTPException detail or log line; encrypted ciphertext is safe to log but we choose not to (no value); the M002 PUT-log discipline (key name + actor_id + previous_value_present only) is preserved and extended to generate.
