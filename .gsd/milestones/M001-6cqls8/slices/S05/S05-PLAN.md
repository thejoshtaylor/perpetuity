# S05: System admin panel

**Goal:** Ship the system admin panel — three new backend endpoints (paginated GET /admin/teams, GET /admin/teams/{id}/members reusing the S03 admin pattern, POST /admin/users/{id}/promote-system-admin with idempotent semantics) gated by `get_current_active_superuser`, plus a `/admin/teams` and `/admin/teams/$teamId` route in the frontend with a confirm dialog wired into the existing `/admin` users list — closes R002 by exercising the role gate end-to-end (system_admin sees all teams, regular user gets 403/redirected).
**Demo:** System admin sees all teams in a paginated list, can view members of any team, and can promote a user to system admin via confirm dialog — non-admins get 403

## Must-Haves

- A system admin signs in, navigates to /admin/teams, sees a paginated list of every team in the system (not just their own memberships), drills into any team to see its members, and from /admin promotes a non-admin user to system_admin via a confirm dialog. A regular user hitting /admin/teams or /admin/teams/{id} is redirected to /. Backend integration tests prove: 200 for system_admin on every endpoint, 403 for non-admin (including non-member of the inspected team), pagination skip/limit, idempotent promote (already-system-admin returns 200 no-op), 404 on missing team or user. Playwright E2E proves the happy path and the 403 redirect on the same setup.

## Proof Level

- This slice proves: integration — real HTTP through FastAPI TestClient + real Postgres for backend; real browser through Playwright for frontend.

## Integration Closure

Consumes from S01: `UserRole` enum, `get_current_active_superuser` dep (already resolves `current_user.role == UserRole.system_admin`), `get_current_user` cookie dep. Consumes from S02: Team model + TeamMember model + GET /teams/{id}/members route shape (TeamMembersPublic). Consumes from S03: `_assert_caller_is_team_member` is NOT reused — admin endpoints bypass membership; `_team_admin_count` not needed (system-admin promotion has no counterpart guard). Produces: three `/api/v1/admin/*` endpoints in a new `app/api/routes/admin.py` module, a `_redact_email`-free admin module (UUIDs only), `/admin/teams` and `/admin/teams/$teamId` frontend routes plus a `PromoteSystemAdminDialog` component wired into the existing `/admin` users data table. Milestone-end-to-end after this slice: every M001 success criterion is provable in a browser — signup → personal team → create team → invite → accept → manage roles → system admin sees all teams. Nothing further required for M001.

## Verification

- Structured INFO logs on every admin action: `admin_teams_listed actor_id=<uuid> skip=<n> limit=<n> count=<n>`, `admin_team_members_listed actor_id=<uuid> team_id=<uuid> count=<n>`, `system_admin_promoted actor_id=<uuid> target_user_id=<uuid> already_admin=<bool>`. Inspection surfaces: `SELECT id, name, is_personal FROM team ORDER BY created_at DESC` reproduces /admin/teams; `SELECT role FROM "user" WHERE id=...` reproduces promote outcome. UUIDs only — no email or team name in logs (matches S03 redaction posture). Failure visibility: HTTP status + detail; route guard rejection produces a 403 with the existing `get_current_active_superuser` "doesn't have enough privileges" detail string, which is part of the contract.
