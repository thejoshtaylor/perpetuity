# S03: system_settings API + dynamic workspace_volume_size_gb + partial-apply shrink

**Goal:** Land the system-wide settings infrastructure: a generic `system_settings` (key TEXT PK, value JSONB) Postgres table, a system_admin-gated `GET/PUT /api/v1/admin/settings[/{key}]` API with per-key validators, and the canonical `workspace_volume_size_gb` key. Implements D015's partial-apply shrink rule — PUT to a smaller value returns 200 with a warnings payload listing affected (user_id, team_id, current size_gb, current usage_bytes); existing `workspace_volume` rows keep their old size_gb (cap divergence allowed); new rows use the new value. Orchestrator's hardcoded `default_volume_size_gb=4` is replaced with a per-request lookup against the new table on each fresh-volume `create_volume` call (existing rows never re-derived).</goal>
<parameter name="successCriteria">PUT /api/v1/admin/settings/workspace_volume_size_gb {value:1} as a system_admin when at least one existing workspace_volume has size_gb=4 → response is 200 with `{value:1, warnings:[{user_id,team_id,size_gb:4,usage_bytes:N}, ...]}`; the affected row is unchanged in DB; a fresh signup's first POST /api/v1/sessions creates a workspace_volume with size_gb=1 and a ~1 GiB .img file. Migration round-trips up/down/up cleanly. Non-admin PUT → 403. Invalid value (negative, >256, wrong type) → 422. Logs emit UUIDs only — never email or full_name.
**Demo:** PUT /api/v1/admin/settings/workspace_volume_size_gb {value:1} as a system_admin when at least one existing workspace_volume has size_gb=4 — response is 200 with `{warnings:[{user_id,team_id,size_gb:4,usage_bytes:N}, ...]}`; the affected row is unchanged in DB; a fresh signup's first POST /api/v1/sessions creates a workspace_volume with size_gb=1 and a ~1 GB .img file.

## Verification

- Runtime signals: structured INFO `system_setting_updated actor_id=<uuid> key=<str> previous_value_present=<true|false>` from the PUT handler; INFO `system_setting_shrink_warnings_emitted key=workspace_volume_size_gb actor_id=<uuid> affected=<n>` when warnings are non-empty; INFO `volume_size_gb_resolved source=<system_settings|fallback> value=<n>` from the orchestrator's new lookup helper on every fresh-volume create. All UUIDs only (MEM134); never email, full_name, or team slug. Inspection surfaces: `SELECT key, value FROM system_settings` is the source of truth for live caps; `SELECT user_id, team_id, size_gb FROM workspace_volume WHERE size_gb > <new_default>` reproduces the shrink warnings payload from the DB. Failure visibility: PUT returns 422 with `{detail: 'invalid_value_for_key', key, reason}` from per-key validators; orchestrator logs WARNING `system_settings_lookup_failed key=workspace_volume_size_gb reason=<class>` and falls back to the boot-time default (4) so a transiently-unreachable system_settings row never blocks fresh provisioning. Redaction: log lines never include the JSONB value verbatim because future settings (e.g. SMTP_PASSWORD if anyone ever stores one there) could carry secrets — log presence/absence and the key name only, never the value.

- [x] **T04: End-to-end demo test — admin PUT triggers partial-apply shrink and next signup gets the new cap** `est:3h`
  Land the slice's demo-truth integration test in `backend/tests/integration/test_m002_s03_settings_e2e.py`, marked `e2e`. Flow:
1. Promote the seeded admin@example.com to system_admin if not already (or use the FIRST_SUPERUSER seeded by prestart) and log in to get the admin cookie jar.
2. Sign up alice@example.com. POST /api/v1/sessions with alice's personal team_id → orchestrator provisions a 4 GiB volume (system_settings is empty so resolve falls back to settings.default_volume_size_gb=4). Assert alice's workspace_volume row has size_gb=4 via a direct SQL fetch through a sibling psql exec (`docker compose exec db psql -U postgres -d app -c "SELECT size_gb FROM workspace_volume WHERE user_id='<alice>'"`).
3. As admin, PUT /api/v1/admin/settings/workspace_volume_size_gb {value: 1}. Assert response is 200 with `value=1` and `warnings` is a non-empty list containing alice's row (user_id, team_id, size_gb=4, usage_bytes=null). Assert log line `system_setting_updated actor_id=<seeded admin uuid> key=workspace_volume_size_gb previous_value_present=false` AND `system_setting_shrink_warnings_emitted key=workspace_volume_size_gb actor_id=<uuid> affected=1` appear in the backend's stdout (`docker logs <backend_container>`). Assert alice's workspace_volume row is unchanged (still size_gb=4) — the partial-apply shrink rule.
4. Sign up bob@example.com. POST /api/v1/sessions with bob's personal team_id → orchestrator provisions a 1 GiB volume (system_settings now governs). Assert bob's workspace_volume row has size_gb=1 via the same psql exec. Assert orchestrator log emits `volume_size_gb_resolved source=system_settings value=1` for bob's provision.
5. WS-attach as bob; run `df -BG /workspaces/<team>` and assert reported total is 1 GiB; run `dd if=/dev/zero of=/workspaces/<team>/big bs=1M count=1100` and assert `No space left on device` in the dd output (the same kernel-cap proof S02 demonstrated, only this time the cap was admin-driven, not env-driven).
6. Idempotent PUT: re-PUT `workspace_volume_size_gb` with value=1; assert 200 with `previous_value_present=true` and warnings now empty (alice still has size_gb=4 but the partial-apply rule still emits the same warnings — actually the rule says WARN whenever size_gb > new_value, so the warnings list is non-empty as long as alice exists. Confirm and assert appropriately).
7. Negative cases: non-admin PUT → 403; PUT with value=300 → 422 invalid_value_for_key; PUT to unknown key `bogus_key` → 422 unknown_setting_key.
8. Log redaction sweep: `docker logs <backend>` and `docker compose logs orchestrator` are scanned for alice@example.com / bob@example.com / their full names → assert ZERO matches (MEM134 invariant carried forward from S02/T04).
