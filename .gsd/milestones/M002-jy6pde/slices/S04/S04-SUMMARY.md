# S04: Idle reaper + multi-session per container + sessions REST surface + scrollback endpoint

**Lands the operational lifecycle layer: process-local AttachMap, background two-phase idle reaper with admin-tunable `system_settings.idle_timeout_seconds`, automatic container reap when the last tmux session dies (volume persists + remounts on next provision per D015), and a backend `GET /api/v1/sessions/{sid}/scrollback` proxy with no-enumeration ownership; the demo proves multi-tmux/single-container filesystem sharing, reaper-driven container shutdown, and re-provision-onto-existing-volume against the live compose stack.**

## What Happened

## What this slice delivered

S04 closes the operational lifecycle layer M002 has been pointing at. The big four pieces:

**1. Orchestrator-side process-local AttachMap (T01).** New module `orchestrator/orchestrator/attach_map.py` exposes `AttachMap` ‚Äî a `dict[str, int]` refcount guarded by a single `asyncio.Lock`, with public surface `register(session_id) -> int`, `unregister(session_id) -> int` (floors at zero AND drops the key entirely), `is_attached(session_id) -> bool`, and `live_session_ids() -> set[str]` (snapshot copy). Refcount instead of bool because cooperative `tmux attach-session` lets two simultaneous WS clients share one tmux session ‚Äî the reaper must treat any positive count as live. Module-level singleton uses lazy-init `get_attach_map()` + `set_attach_map(None)` setter (mirrors `set_registry`/`set_pool`). No external resource ‚Üí no lifespan binding needed; routes_ws.py is importable in unit suites without a running lifespan. The map is INTENTIONALLY in-process ‚Äî an orchestrator restart drops every WS attach because the exec stream dies (D012: tmux survives, attach does not), so a restart-time empty map is the right truth. `routes_ws.py::session_stream` is instrumented: `register(session_id)` is called AFTER `await stream.__aenter__()` succeeds (so a failed exec start never bumps the refcount) and `unregister(session_id)` lives in a `finally` block wrapping the entire pump-and-teardown section so any mid-stream raise still decrements. Both calls emit `attach_registered`/`attach_unregistered session_id=<uuid> count=<n>` INFO lines (UUIDs only per MEM134).

**2. Background idle reaper with two-phase liveness check (T02).** New module `orchestrator/orchestrator/reaper.py` exposes `start_reaper(app)` / `stop_reaper(task)` and `reaper_loop`. Wired into `main.py::_lifespan` AFTER Redis registry, attach map, and pg pool are bound; `app.state.reaper_task` is set to the task handle. `stop_reaper` runs FIRST in the lifespan teardown (before registry.close / close_pool / docker.close) with a 5s teardown budget so an in-flight tick doesn't trip on torn-down resources. The loop sleeps `REAPER_INTERVAL_SECONDS` (env-overridable, default 30s, clamp [1, 300]), then on each tick: (a) resolves idle timeout via the new `volume_store._resolve_idle_timeout_seconds(pool)` helper (mirrors `_resolve_default_size_gb` exactly ‚Äî same SELECT, JSONB-as-text local json.loads, bool-rejection, range gate [1, 86400], fallback to `settings.idle_timeout_seconds` on miss/invalid/error with WARNING `system_settings_lookup_failed`); (b) iterates `registry.scan_session_keys()` ‚Äî a NEW `AsyncIterator` on `RedisSessionRegistry` that wraps `SCAN MATCH session:*` (production-hostile `KEYS` is avoided); (c) D018 two-phase check ‚Äî reapable iff `idle > timeout AND not attach_map.is_attached(sid)`; (d) `kill_tmux_session` (idempotent) then `delete_session` from Redis; (e) for any container the reaper just emptied, double-checks `list_tmux_sessions == []` AND `_find_container_by_labels` still resolves to the SAME container_id (label-collision guard against a fresh re-provision in the same (user, team) between the kill pass and the reap pass), then `container.stop(timeout=5)` + `container.delete(force=True)`. The workspace_volume row + underlying `.img` are NEVER touched ‚Äî D015 invariant: next POST refinds the row and re-mounts via `ensure_volume_for` (idempotent). Every tick is wrapped in `try/except Exception` logging WARNING `reaper_tick_failed reason=<class>` so transient errors never kill the task; only `asyncio.CancelledError` exits the loop. Mid-execution gotcha (MEM168): `sessions.py` wraps DockerError + OSError into `DockerUnavailable` at the boundary, so the reaper has to catch DockerUnavailable too at every Docker-touching call site ‚Äî without this, every tick that touches a missing/gone container surfaces as `reaper_tick_failed reason=DockerUnavailable` and no session ever reaps. Backend admin: `idle_timeout_seconds` validator added to `_VALIDATORS` (bool rejected, int required, range [1, 86400], 422 `invalid_value_for_key`); no partial-apply warnings (the new value just biases the next reaper tick); constant `IDLE_TIMEOUT_SECONDS_KEY` for symmetry.

**3. Backend public scrollback proxy (T03).** `GET /api/v1/sessions/{session_id}/scrollback` in `backend/app/api/routes/sessions.py` ‚Äî proxies the orchestrator's existing `POST /v1/sessions/{sid}/scrollback`. Public verb is GET (read should be a GET so the FE doesn't POST-for-read) even though the orchestrator endpoint stays POST per S01's locked plan; the asymmetry is documented in the route docstring. Response shape is `{session_id: str, scrollback: str}` ‚Äî the raw UTF-8 string from the orchestrator, NOT base64 (only the WS attach frame base64-encodes scrollback per the locked frame protocol). Ownership reuses `_orch_get_session_record(session_id)` from sessions.py ‚Äî same no-enumeration rule as DELETE: missing record OR record.user_id != current_user.id ‚Üí 404 `{"detail": "Session not found"}` byte-equal across the two cases (MEM113/MEM123). Orchestrator unreachable ‚Üí 503 via `_orch_unavailable_503`. Schema-drift safety net: if the orchestrator response is not a dict, lacks the `scrollback` key, or returns a non-string value, raise 503 instead of crashing with KeyError ‚Äî converts future orchestrator-side schema drift into an actionable 503 rather than a 500. INFO `session_scrollback_proxied session_id=<uuid> user_id=<uuid> bytes=<n>` on success ‚Äî content is NEVER logged (could be echoed secrets), only byte length. Test infrastructure: a `_FakeAsyncClient` + `_install_fake_orch` helper monkeypatches `app.api.routes.sessions.httpx.AsyncClient` so each test scripts orchestrator responses for lookup and fetch independently ‚Äî no respx dev-dep added; reusable for future orchestrator-proxy routes (MEM172).

**4. Demo-truth e2e (T04).** `backend/tests/integration/test_m002_s04_e2e.py::test_m002_s04_full_demo` (e2e marker, ~19s wall-clock against real Postgres + Redis + orchestrator + Docker daemon). The orchestrator is swapped in-place via the S02 live-orchestrator-swap pattern (MEM149) to inject `REAPER_INTERVAL_SECONDS=1` since compose's orchestrator service has no env hook for it; fixture restores the compose orchestrator on teardown. Single async test drives the full S04 demo: admin PUTs `idle_timeout_seconds=600` up-front (so prep steps don't race a 1s reaper tick); alice signs up and POSTs two sessions for her personal team ‚Äî orchestrator response.created==True for the first, False for the second (T03/MEM120 ‚Äî same container reused, distinct tmux sessions inside); WS-attach to sid_a writes `'a' > /workspaces/<team_id>/marker.txt`, WS-attach to sid_b reads `cat /workspaces/<team_id>/marker.txt` and ANSI-strips for `'a'` in the data-frame stream (proves R008 multi-tmux/single-container filesystem sharing); GET /api/v1/sessions returns set {sid_a, sid_b}; GET /api/v1/sessions/{sid_a}/scrollback returns 200 with the marker (T03 happy path); fresh signup `bob` GETs sid_a's scrollback and a never-existed UUID ‚Äî both return 404 with byte-equal bodies (no-enumeration); DELETE sid_a leaves sid_b AND the container alive; admin PUTs `idle_timeout_seconds=3` (two-phase PUT pattern ‚Äî MEM175); sleep 6s; GET /api/v1/sessions returns empty; `docker ps --filter label=...` returns zero rows (container reaped); alice POSTs a third session, WS-attaches, reads the SAME marker ‚Äî proving the workspace_volume row + .img persisted across the reap and the new container remounted the existing volume (D015 invariant + R006 + the slice success criterion). MEM134 redaction sweep across `docker compose logs orchestrator backend` finds zero alice/bob email or full_name matches across all reaper/attach/scrollback log lines. Asserted log lines: `reaper_killed_session reason=idle_no_attach`, `reaper_reaped_container reason=last_session_killed`, `attach_registered`, `attach_unregistered`, `idle_timeout_seconds_resolved`, `session_scrollback_proxied`. Two autouse fixtures harden the test: a stale-image probe that skips with `docker compose build backend` guidance when the new T03 scrollback route isn't baked in (MEM173 ‚Äî cost ~30 min in dev to a confusing FastAPI default 404 before this was added); and a `system_settings` DELETE-before-and-after for `idle_timeout_seconds` per MEM161.

## Patterns established for downstream slices

- **D018 two-phase liveness check** ‚Äî every background reaper-style task in this codebase should AND together a Redis-side staleness signal (`last_activity > timeout`) and an in-process liveness signal (AttachMap.is_attached). Either half alone is insufficient: Redis-only kills active-but-quiet sessions; AttachMap-only never reaps after restart since the map empties.
- **Process-local refcount + asyncio.Lock pattern** ‚Äî for any orchestrator-internal liveness counter that is correctly invalidated by orchestrator restart. Lazy-init singleton accessor (no lifespan binding required).
- **Reaper loop discipline** ‚Äî `try/except Exception` around every iteration so transients log WARNING `<task>_tick_failed reason=<class>` and continue; `asyncio.CancelledError` is the only legitimate exit; `stop_<task>` runs FIRST in lifespan teardown before its dependencies close.
- **DockerUnavailable catch contract (MEM168)** ‚Äî every background caller of `orchestrator/orchestrator/sessions.py` MUST catch DockerUnavailable in addition to DockerError + OSError because sessions.py wraps both at the boundary.
- **Backend orchestrator-proxy schema-drift safety net** ‚Äî proxy routes raise 503 `orchestrator_unavailable` on missing/wrong-typed response keys rather than crash with KeyError. Costs nothing on the happy path; converts future orchestrator schema drift into an actionable 503.
- **Backend‚Üîorchestrator HTTP-verb asymmetry** ‚Äî public verb follows REST semantics (GET for reads); internal verb stays whatever was settled at locked-plan time. Document in the route docstring.
- **`_FakeAsyncClient` test pattern (MEM172)** ‚Äî backend orchestrator-proxy routes get co-located fake-AsyncClient helpers per test module rather than a global respx dependency. Reusable for any future orchestrator-proxy route.
- **REAPER_INTERVAL_SECONDS live-orchestrator-swap (MEM188)** ‚Äî e2e tests that need a faster reaper than the compose default use the S02 swap pattern, NOT a `docker-compose.override.yml`. No test-only file to clean up; teardown restores the compose orchestrator.
- **Two-phase idle_timeout PUT in e2e tests (MEM175)** ‚Äî first PUT `idle_timeout_seconds=600` so prep doesn't race the reaper; second PUT to the test value (e.g. 3) right before the reap-wait sleep.
- **Stale-backend-image autouse skip-guard (MEM173)** ‚Äî every M002+ e2e depending on a freshly-baked backend feature ships an autouse probe over `backend:latest` that skips with a `docker compose build backend` instruction on miss. Mirrors S03's alembic-revision skip-guard (MEM162).
