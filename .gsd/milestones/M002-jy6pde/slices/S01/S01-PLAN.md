# S01: Orchestrator service + tmux-durable WS terminal (the proof-first slice)

**Goal:** Establish the orchestrator service, Redis session registry, tmux-durable terminal sessions, and the cookie-authed backend WS bridge so that a signed-up user can connect to `wss://<host>/api/v1/ws/terminal/<session_id>`, run `echo hello`, restart the orchestrator, and reconnect to the SAME session_id with prior scrollback intact and the same shell PID. This slice locks the WS frame protocol contract (server: attach/data/exit/detach; client: input/resize) for the lifetime of M002+. Backend-only delivery — no frontend.
**Demo:** Signup as a fresh user via M001 endpoints, then open a WS to /api/v1/ws/terminal/<new-uuid> with the session cookie. Observe `{type:"attach",scrollback:""}` then send `{type:"input",bytes:"echo hello\n"}` and receive `{type:"data",bytes:"hello\r\n"}` (or equivalent line). Run `docker compose restart orchestrator`. Reconnect to the SAME session_id; observe `{type:"attach",scrollback:"...hello..."}` and confirm `echo $$` returns a stable shell PID across reconnects. This is exercised end-to-end by an automated integration test against the real compose stack — no mocks.

## Must-Haves

- A signed-up user with a personal team can connect via WS to `/api/v1/ws/terminal/<new_session_id>` (after `POST /api/v1/sessions`), receive `{type:"attach",scrollback:""}`, send `{type:"input",bytes:"echo hello\n"}`, and receive `{type:"data",bytes}` containing `hello`. After `docker compose restart orchestrator`, reconnecting to the SAME session_id yields `{type:"attach",scrollback:"...hello..."}` AND `echo $$` returns the same shell PID as before the restart (confirming the tmux session — not a new shell — is what survived). All proven by an integration test running against the real compose stack with real Postgres + real Redis + real Docker daemon — zero mocks of those boundaries. Cookie auth failures close 1008 (matching M001). Shared-secret failures on the orchestrator hop close 1008 with reason `unauthorized`. Session ownership violations (user B attaches to user A's session_id, or to a never-existed session_id) close 1008 `session_not_owned` identically — no existence enumeration. Redis or Docker daemon unreachable propagates as 503. All sensitive log lines emit user/team/session/container IDs as UUIDs only — never email or full name.

## Proof Level

- This slice proves: Integration. The contract under test crosses three runtime boundaries (browser→backend WS, backend→orchestrator WS, orchestrator→docker exec→tmux) and the orchestrator-restart proof is fundamentally non-mockable: it requires a real tmux process in a real container surviving a real `docker compose restart orchestrator`. The slice fails verification if any test substitutes aiodocker, Redis, or Postgres with a mock. CI mounts the host Docker socket and uses `perpetuity/workspace:test` to keep image-pull cost bounded.

## Integration Closure

Slice integration is closed when (a) `docker compose up` brings up backend + orchestrator + redis + db all healthy from scratch on a clean machine; (b) the end-to-end integration test in T06 passes against that real stack including the orchestrator-restart reconnect step; (c) the WS frame protocol shape (attach/data/exit/detach/input/resize) is locked in `orchestrator/orchestrator/protocol.py` and `backend/app/api/routes/sessions.py` — downstream slices S02–S05 may add fields under existing types but must not change frame `type` values; (d) the workspace image `perpetuity/workspace:latest` (and `perpetuity/workspace:test`) is buildable from `orchestrator/workspace-image/Dockerfile` and contains tmux, bash, git, node, python; (e) shared-secret two-key acceptance is wired in code (S05 hardens the rotation acceptance test, but the orchestrator code path that accepts either key must already exist and be unit-tested in S01).

## Verification

- This slice introduces the M002 observability taxonomy from the milestone CONTEXT appendix. Required INFO log keys emitted in S01: `session_created`, `session_attached`, `session_detached`, `image_pull_ok`, `container_provisioned`. WARNING: `redis_unreachable`, `docker_unreachable`, `tmux_session_orphaned`. ERROR: `image_pull_failed` (boot blocker), `orchestrator_ws_unauthorized`. All log lines that include user/team/session/container identifiers MUST emit UUIDs only — never email, full_name, or team slug. Backend WS auth failures continue using the existing M001 taxonomy (`ws_auth_reject reason=...`). A regression test in T06 greps the test-run log capture for any occurrence of the seeded test user's email or full_name and fails if found.

- [x] **T04: Orchestrator WS bridge: bidirectional bytes between WS and tmux exec** `est:XL`
  Implement the orchestrator-side WS endpoint that streams bytes between an authenticated client and a running tmux session. In `orchestrator/orchestrator/routes_ws.py`: `WS /v1/sessions/{session_id}/stream` — auth via shared secret (T02 query-param pattern). On connect: (1) read Redis session record; if missing → close 1008 reason=`session_not_found`. (2) Send `{type:'attach', scrollback: <100KB-capped>}` JSON frame to client. (5) On WS client disconnect → cancel pumps but do NOT kill tmux session (the whole point of D012 — tmux stays alive). (6) Update Redis `last_activity` on every input frame received (this is the heartbeat S04's reaper depends on). Frame protocol locked here — JSON-encoded UTF-8, `data` and `input` payloads are base64-encoded raw bytes (handles binary, locale, escape sequences cleanly). Add a `protocol.py` module that exports the typed frame schemas (TypedDict or pydantic) — both backend and orchestrator import from it eventually, but for S01 the canonical home is `orchestrator/orchestrator/protocol.py` and the backend gets a copy in T05. ASSUMPTION (auto-mode): WS message size limit is 1 MB on the FastAPI side; 100 KB scrollback frames are well under. ASSUMPTION: base64-encoding `bytes` payloads adds ~33% overhead but is the only way to round-trip binary safely in JSON — accepted.
