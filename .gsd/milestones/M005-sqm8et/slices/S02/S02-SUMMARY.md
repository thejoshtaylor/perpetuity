# S02: Dashboard direct AI buttons (Claude + Codex) — proves the AI executor end-to-end

**Shipped the load-bearing AI executor spine — minimal workflows/workflow_runs/step_runs schema, auto-seeded `_direct_claude` / `_direct_codex` system workflows, orchestrator one-shot exec endpoint with util-linux script(1) TTY discipline, Celery worker + ai executor, `POST /workflows/{id}/run` + `GET /workflow_runs/{id}` + dashboard buttons + 1.5s-polled run-detail page — all proven end-to-end against the live compose stack via a test-shim CLI.**

## What Happened

S02 retired the load-bearing technical risk in M005: that the Celery → orchestrator HTTP → docker exec → `script(1)` wrapped CLI chain would not actually work end-to-end. By the close of this slice every link in that chain has been proven against the live compose stack with a deterministic test-shim CLI, and the surfaces S03/S04/S05 build on (workflow schema, AI step executor, run/step record shape, polling API + UI) are locked.

**T01 — Schema spine.** Two new alembic revisions (`s10_workflows`, `s11_workflow_runs`) introduce four tables shaped to accommodate S03's eventual multi-step + scope + per-step `target_container` so S03 doesn't migrate. `workflows` (UUID PK, team FK CASCADE, UNIQUE(team_id,name), `system_owned BOOLEAN` for D028 filtering, `scope` 3-valued CHECK), `workflow_steps` (sibling table — not JSONB on workflows so S03 can ALTER without rewriting; UNIQUE(workflow_id, step_index); `action` 4-valued CHECK covering claude/codex/shell/git with the latter two reserved for S03; JSONB `config`), `workflow_runs` (CASCADE on workflow + team, SET NULL on triggered_by/target_user so user delete preserves the audit trail, `trigger_type`/`status` CHECK enums, JSONB `trigger_payload`, `last_heartbeat_at` reserved for S05's recovery, `error_class` as free-text VARCHAR(64) so S03/S04/S05 layer in `webhook_validation_failed` / `dispatch_failed` / etc. without an ALTER), and `step_runs` (CASCADE on workflow_run, JSONB `snapshot` of WorkflowStep at dispatch time, persisted `stdout`/`stderr` per R018 with TEXT NOT NULL DEFAULT '', `exit_code`/`error_class`/`duration_ms`, UNIQUE(workflow_run_id, step_index)). Eleven new SQLModel rows + DTOs in `app/models.py`; per MEM352, enum columns use Python Enum classes (`WorkflowScope`, `WorkflowAction`, `WorkflowRunTriggerType`, `WorkflowRunStatus`, `StepRunStatus`) so OpenAPI emits string-literal unions to the frontend. 20 round-trip migration tests cover column shape + FK semantics + UNIQUE/CHECK rejections + cascade through every edge.

**T02 — TTY discipline + system seeds.** Three deliverables in one transaction: (a) `orchestrator/workspace-image/Dockerfile` pins `@anthropic-ai/claude-code@1.0.30` and `@openai/codex@0.20.0` via `npm install -g`, with TTY smoke checks (`script -q -e -c '<cli> --version' /dev/null`) so CLI regressions on TTY discipline fail the image build; (b) `orchestrator/orchestrator/routes_exec.py` is the new `POST /v1/sessions/{session_id}/exec` route — pydantic caps on cmd/env entry counts and lengths, shell-quotes cmd via `shlex.quote` EXCEPT bare `$NAME`/`${NAME}` references which are passed as `"$NAME"` so `/bin/sh` (invoked by `script -c`) expands them against the env dict (MEM274 secret-passing pattern: API keys flow only through the env frame, never the cmd list), `_exec_collect_with_env` opens docker exec with `tty=True` and drains stdout up to 5 MiB heap cap, `asyncio.wait_for` surfaces a clean 504 `oneshot_exec_timeout` rather than a wedged exec stream; (c) `s12_seed_direct_workflows.py` data migration backfills `_direct_claude` and `_direct_codex` workflows for every existing team (`system_owned=TRUE`, `scope='user'`, one step at index 0 with config `{"prompt_template": "{prompt}"}`), and `seed_system_workflows(session, team_id)` in `workflows_seed.py` is wired into BOTH `crud.py` team-create paths (`create_user_with_personal_team`, `create_team_with_admin`) BEFORE `session.commit()` so seed failures roll the team back. **Deviation captured as MEM427:** the inlined plan said wrap as BSD `script -q /dev/null sh -c '<body>'`; Ubuntu 24.04 ships util-linux script which uses `script [opts] [file]` and the BSD form errors with `script: unexpected number of arguments`. Switched to `script -q -e -c '<body>' /dev/null`. The `-e` (`--return`) flag is critical — without it script always exits 0 regardless of child exit code, masking CLI failures; the non-zero-exit integration test caught this.

**T03 — Worker spine + AI executor.** `backend/app/core/celery_app.py` is a single `Celery("perpetuity")` instance bound to the existing Redis broker, no result backend (Postgres `workflow_runs.status` is the SoT per MEM009), with `task_acks_late=True` + `task_reject_on_worker_lost=True` configured now so S05's orphan-recovery beat doesn't have to retroactively change worker config, and `worker_prefetch_multiplier=1` because per-step cost is dominated by the orchestrator HTTP call. `backend/app/workflows/executors/ai.py::run_ai_step` owns the per-step lifecycle for `claude` and `codex` actions: reads team API key via `get_team_secret` (S01 helper), renders `snapshot.config.prompt_template` with `{prompt}` substituted from `WorkflowRun.trigger_payload['prompt']` using `str.replace` (NOT `str.format` — defends against user prompts containing `{` characters), POSTs the orchestrator's exec endpoint with API key + prompt body in the env dict, maps the failure space to the slice's error_class taxonomy (`missing_team_secret`, `team_secret_decrypt_failed`, `orchestrator_exec_failed`, `cli_nonzero`). `derive_session_id(target_user_id, team_id, run_id)` returns a deterministic UUID5 (MEM429) so re-runs and Celery double-deliveries hit the same workspace container. `backend/app/workflows/tasks.py::run_workflow(run_id)` is the runner spine: idempotency-guards on `status != 'pending'`, transitions pending→running with `started_at` + `last_heartbeat_at`, iterates WorkflowStep rows in step_index order, dispatches by action, propagates the failed step's error_class to the parent run, marks succeeded only if every step succeeded; wrapper opens a FRESH session for a best-effort `error_class='worker_crash'` stamp on unhandled exceptions then re-raises. 20 unit tests across `test_workflow_executor_ai.py` + `test_workflow_runner.py` cover deterministic session_id, claude+codex happy paths, all four executor failure modes, prompt template substitution, idempotency, error_class propagation, snapshot freezing per R018, Celery task registration.

## Deviations

**T02 — script(1) syntax (MEM427).** The inlined task plan said wrap as BSD `script -q /dev/null sh -c '<body>'`. Ubuntu 24.04 ships util-linux script which uses `script [opts] [file]` — the BSD form errors with `script: unexpected number of arguments`. Corrected to `script -q -e -c '<body>' /dev/null` and updated both the orchestrator wrapper (`_build_script_cmd`) and the Dockerfile smoke check. The `-e` flag was an additional hidden requirement — without it script always exits 0, masking CLI failures. Captured as MEM427.

**T03 — get_datetime_utc import location.** Plan implied `app.utils.get_datetime_utc`; the helper actually lives in `app.models`. Imported from there. No semantic difference.

**T03 — Prompt template substitution.** Plan wording said "substituting `{prompt}`". Implemented as `str.replace('{prompt}', value)` rather than `str.format(prompt=value)` so a user prompt containing `{` characters can't trip a KeyError or leak format-spec features. Matches plan intent.

**T04 — Membership helper detail translation.** Plan said use `assert_caller_is_team_member` directly. The shared helper raises `HTTPException(detail="...")` with free-text strings; the slice plan locks structured `{detail: "<discriminator>"}` shapes. Caught and re-raised at each call site rather than forking a second helper — keeps the membership boundary single-sourced while routes own the response shape.

**T05 — Membership gate placement.** Plan asked to gate DirectAIButtons on 'at least member' role. Rendered inside `TeamDetailContent` after the team-not-found short-circuit — that gate IS the membership check since the route only renders when the user's `readTeams()` envelope contains the URL teamId. Admin-only restriction would gate the demo against the most common caller.

**Verification command form.** Slice plan verify command uses `cd backend && pytest ...` chains. The GSD verification runner naively splits on `&&` and runs each segment as a separate process, so `cd backend` is a no-op for the next process and pytest fails with "file or directory not found" — this caused the auto-fix retry signal that prompted this slice closure. Re-running with explicit `cd /Users/josh/code/perpetuity/backend && POSTGRES_DB=perpetuity_app uv run pytest -m e2e tests/integration/test_m005_s02_dashboard_ai_buttons_e2e.py -v` succeeds in a single shell. Same harness behavior bit S01's first close.

## Known Limitations

- **Real Anthropic / real OpenAI not exercised.** Per D029, S02 uses deterministic in-container test-shim CLIs. Real-API acceptance is reserved for S06 against the pinned `@anthropic-ai/claude-code@1.0.30` and `@openai/codex@0.20.0`. The TTY discipline is image-build-asserted so a real-API regression on TTY semantics fails the build.
- **No retry/backoff for orchestrator HTTP failures.** S02 surfaces `orchestrator_exec_failed` directly on the first httpx error. S03 plan calls for "retries container acquisition 3x exponential" — the seam is in `run_ai_step`'s httpx call site.
- **No cancellation between steps.** S03 owns this. S02 has exactly one step per run (`_direct_claude` / `_direct_codex` are single-step workflows) so the cancellation primitive is not yet exercised.
- **No worker crash recovery.** `last_heartbeat_at` is stamped on pending→running transition but no Beat task watches it — that's S05's `recover_orphan_runs` Beat. A worker crash mid-run will leave the run in `running` until S05 lands.
- **No operational caps.** `max_concurrent_runs` and `max_runs_per_hour` are deferred to S05. S02 dispatch is unguarded — concurrent triggers all execute. The S05 plan mounts caps at the API trigger boundary where `dispatch_failed` already exists.
- **Run history list page absent.** GET /runs?team_id=... + filters (status, trigger type, time range) are deferred to S05. Only the per-run drill-down (`GET /workflow_runs/{id}`) is shipped.
- **Admin manual trigger absent.** `POST /admin/workflows/{id}/run` deferred to S05. S02 only ships member-role-gated `POST /workflows/{id}/run`.
- **`script -e` exit-code propagation has only been verified for the immediate exit code of the command body.** If `claude` / `codex` daemonize a child or trap SIGTERM during a future timeout, behavior is undefined. T03 known issue carried forward — S03 should add an integration test that covers a CLI killed mid-stream.
- **`worker_crash` discriminator stamp is best-effort.** The wrapper opens a fresh session (the original may be in aborted-transaction state after the unhandled exception), but if the fresh session also fails, the crash will manifest as a stuck `running` run that S05's recovery beat resolves.
- **Stdout 5 MiB heap cap on orchestrator side.** A pathological CLI that streams 100+ MiB of output will be truncated. backend's `step_runs.stdout` column accepts any size but realistic shape is far below this.
- **System workflow seed payload duplicated** between `workflows_seed.SYSTEM_WORKFLOWS` (runtime) and `s12_seed_direct_workflows._SYSTEM_WORKFLOWS` (migration). Future changes to the prompt_template or step shape need to land in both places. MEM428 captured the rationale.
- **Cosmetic SDK regen lint failure.** `scripts/generate-client.sh`'s trailing `bun run lint` fails on a pre-existing biome a11y rule in voice components (unrelated to this slice). The SDK files are regenerated correctly before that step. Future regens may need a one-time biome fix or `--no-lint` toggle.
